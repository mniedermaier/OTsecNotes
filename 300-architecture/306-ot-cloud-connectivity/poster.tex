\section{\textcolor{accent}{\faIcon{cloud}}\hspace{0.4em}Overview}

Cloud connectivity for OT should be \textbf{highly avoided}. When business requirements make it unavoidable, connectivity must be strictly limited to \textbf{outbound data flows for monitoring only}. The default position is no cloud connectivity.

\posterdanger{
\faIcon{ban}\hspace{0.2em}\textbf{Absolute Rule:} Remote control, command execution, or \textbf{any data flow from cloud into OT control systems is unacceptable}. No exceptions. If a use case requires cloud-to-OT control, it must be rejected or redesigned.
}

\subsection{\textcolor{accent}{\faIcon{long-arrow-alt-right}}\hspace{0.3em}Pattern 2: Data Diode + Cloud}

\begin{itemize}
    \item \faIcon{long-arrow-alt-right}\hspace{0.2em}Hardware-enforced unidirectional from OT

\section{\textcolor{accent}{\faIcon{exclamation-triangle}}\hspace{0.4em}Security Challenges}

\begin{enumerate}
    \item \faIcon{expand-arrows-alt}\hspace{0.2em}Expanded attack surface
    \item \faIcon{handshake}\hspace{0.2em}Cloud provider as trusted third party
    \item \faIcon{flag}\hspace{0.2em}Data sovereignty and compliance
    \item \faIcon{key}\hspace{0.2em}Credential management complexity
    \item \faIcon{eye-slash}\hspace{0.2em}Visibility gaps in encrypted traffic
    \item \faIcon{wifi}\hspace{0.2em}Dependency on internet availability
\end{enumerate}

\section{\textcolor{accent}{\faIcon{shield-alt}}\hspace{0.4em}Security Controls}

\subsection{\textcolor{accent}{\faIcon{network-wired}}\hspace{0.3em}Network Security}

\begin{itemize}
    \item \faIcon{long-arrow-alt-right}\hspace{0.2em}\textbf{Outbound-only} -- OT/edge initiates all connections
    \item \faIcon{list-alt}\hspace{0.2em}\textbf{Whitelist} -- Specific cloud endpoints only
    \item \faIcon{lock}\hspace{0.2em}\textbf{TLS 1.3} -- Encrypt all data in transit
    \item \faIcon{certificate}\hspace{0.2em}\textbf{Certificate pinning} -- Prevent MitM attacks
    \item \faIcon{bell}\hspace{0.2em}\textbf{Monitor} -- Alert on unexpected destinations
\end{itemize}

\subsection{\textcolor{accent}{\faIcon{user-lock}}\hspace{0.3em}Identity and Access}

\begin{itemize}
    \item \faIcon{certificate}\hspace{0.2em}Unique certificates per edge gateway
    \item \faIcon{robot}\hspace{0.2em}Managed identities (no stored credentials)
    \item \faIcon{key}\hspace{0.2em}Least privilege cloud permissions
    \item \faIcon{fingerprint}\hspace{0.2em}MFA for all human access to cloud OT data
    \item \faIcon{sync}\hspace{0.2em}Automated credential rotation
\end{itemize}

\faIcon{compress-arrows-alt}\hspace{0.2em}Apply \textbf{data minimization}: only send data actually needed. Filter, aggregate, and anonymize at the edge. The edge gateway is a critical security boundary.
}

\section{\textcolor{accent}{\faIcon{clipboard-check}}\hspace{0.4em}Implementation Checklist}

\begin{enumerate}
    \item \faIcon{file-alt}\hspace{0.2em}Document business justification
    \item \faIcon{ban}\hspace{0.2em}Verify no cloud-to-OT control requirement
    \item \faIcon{tags}\hspace{0.2em}Classify data sensitivity before upload
    \item \faIcon{sitemap}\hspace{0.2em}Select architecture pattern per risk level
    \item \faIcon{long-arrow-alt-right}\hspace{0.2em}Implement outbound-only connectivity
    \item \faIcon{server}\hspace{0.2em}Deploy edge gateway in DMZ
    \item \faIcon{filter}\hspace{0.2em}Configure firewalls with minimal endpoints
    \item \faIcon{certificate}\hspace{0.2em}Establish device identity and credential mgmt
    \item \faIcon{lock}\hspace{0.2em}Enable encryption in transit and at rest
    \item \faIcon{eye}\hspace{0.2em}Implement monitoring and alerting
\end{enumerate}

\postertip{
\faIcon{lightbulb}\hspace{0.2em}Edge gateway in DMZ with \textbf{outbound-only firewall rules} is the recommended pattern. Block all inbound from cloud. OT systems should \textbf{never depend on internet availability} for safe operation.
}
