\section{\textcolor{accent}{\faIcon{exclamation-circle}}\hspace{0.4em}Overview}

The Purdue Enterprise Reference Architecture (1990s) guided OT segmentation for decades. It assumes \textbf{clear physical boundaries} between hierarchical levels with traffic flowing between adjacent zones. Modern technologies -- cloud, SDN, virtualization, IIoT -- challenge these fundamental assumptions.

\posterdanger{
\faIcon{bolt}\hspace{0.2em}Cloud, IIoT gateways, and vendor VPNs create \textbf{direct paths from L0--L2 to the internet}, bypassing DMZ and enterprise layers. A compromised cloud account can reach control systems directly.
}

\subsection{\textcolor{accent}{\faIcon{network-wired}}\hspace{0.3em}SDN \& Converged Infrastructure}

\begin{itemize}
    \item \faIcon{code-branch}\hspace{0.2em}\textbf{Virtual switches} -- VM traffic may not traverse FW
    \item \faIcon{layer-group}\hspace{0.2em}\textbf{Overlay networks} -- Logical nets span physical infra
    \item \faIcon{docker}\hspace{0.2em}\textbf{Containers} -- Dynamic IPs break static FW rules
    \item \faIcon{server}\hspace{0.2em}\textbf{Hyperconverged} -- Multiple levels on single host
    \item \faIcon{microchip}\hspace{0.2em}\textbf{Virtual PLCs} -- Control logic mixed with IT workloads
    \item \faIcon{database}\hspace{0.2em}\textbf{Unified namespace} -- Data published across all levels
\end{itemize}

\section{\textcolor{accent}{\faIcon{exclamation-triangle}}\hspace{0.4em}Security Implications}

\subsection{\textcolor{accent}{\faIcon{crosshairs}}\hspace{0.3em}Attack Surface Expansion}

\begin{itemize}
    \item \faIcon{cloud}\hspace{0.2em}Cloud credentials compromise reaches OT directly
    \item \faIcon{random}\hspace{0.2em}Lateral movement bypasses zone firewalls
    \item \faIcon{truck}\hspace{0.2em}Supply chain attacks through connected vendors
    \item \faIcon{globe}\hspace{0.2em}Internet-exposed OT components become targets
\end{itemize}

\subsection{\textcolor{accent}{\faIcon{eye-slash}}\hspace{0.3em}Visibility Gaps}

\begin{itemize}
    \item \faIcon{arrows-alt-h}\hspace{0.2em}East-west traffic in virtualized environments invisible
    \item \faIcon{lock}\hspace{0.2em}Encrypted cloud connections hide content
    \item \faIcon{code}\hspace{0.2em}API calls don't match traditional FW rules
    \item \faIcon{sync}\hspace{0.2em}Dynamic workloads evade static monitoring
\end{itemize}

\subsection{\textcolor{accent}{\faIcon{layer-group}}\hspace{0.3em}Defense in Depth Layers}

When perimeter controls weaken, layer additional defenses:

\begin{enumerate}
    \item \faIcon{id-card}\hspace{0.2em}Identity and Access Management
    \item \faIcon{project-diagram}\hspace{0.2em}Network Segmentation (where possible)
    \item \faIcon{shield-alt}\hspace{0.2em}Endpoint Protection
    \item \faIcon{code}\hspace{0.2em}Application Security
    \item \faIcon{lock}\hspace{0.2em}Data Protection and Encryption
    \item \faIcon{eye}\hspace{0.2em}Monitoring and Detection
\end{enumerate}

\subsection{\textcolor{accent}{\faIcon{fingerprint}}\hspace{0.3em}Identity-Centric Security}

\begin{itemize}
    \item \faIcon{fingerprint}\hspace{0.2em}\textbf{Strong auth} -- MFA for all remote/privileged access
    \item \faIcon{robot}\hspace{0.2em}\textbf{Service accounts} -- Managed machine-to-machine IDs
    \item \faIcon{clock}\hspace{0.2em}\textbf{Just-in-time} -- Temporary privileges for maintenance
    \item \faIcon{certificate}\hspace{0.2em}\textbf{Certificate-based} -- Device identity for OT endpoints
\end{itemize}

\subsection{\textcolor{accent}{\faIcon{chart-line}}\hspace{0.3em}Enhanced Monitoring}

\begin{itemize}
    \item \faIcon{brain}\hspace{0.2em}\textbf{Behavioral analytics} -- Detect anomalies within zones
    \item \faIcon{code}\hspace{0.2em}\textbf{API monitoring} -- Track cloud and integration traffic
    \item \faIcon{user-check}\hspace{0.2em}\textbf{Identity analytics} -- Unusual access patterns
    \item \faIcon{lock}\hspace{0.2em}\textbf{Encrypted traffic} -- Metadata-based detection
\end{itemize}

\section{\textcolor{accent}{\faIcon{clipboard-check}}\hspace{0.4em}Practical Recommendations}

\posterwarning{
\faIcon{search}\hspace{0.2em}\textbf{Assessment first:} Map actual traffic flows, not assumed architecture. Identify all cloud connections and remote access paths. Document where virtualization spans zones.
}

\begin{itemize}
    \item \faIcon{shield-alt}\hspace{0.2em}Enforce strict segmentation for safety systems
    \item \faIcon{tools}\hspace{0.2em}Use compensating controls where boundaries are weak
    \item \faIcon{crosshairs}\hspace{0.2em}Prioritize \textbf{critical asset protection} over zone purity
    \item \faIcon{file-alt}\hspace{0.2em}Document all exceptions and accepted risks
    \item \faIcon{sync}\hspace{0.2em}Re-evaluate architecture annually as technology evolves
    \item \faIcon{sitemap}\hspace{0.2em}Map actual data flows -- not assumed topology
\end{itemize}

\postersuccess{
\faIcon{check-circle}\hspace{0.2em}\textbf{Hybrid approach:} Use the Purdue Model as a conceptual starting point but augment with Zero Trust, identity-centric controls, and continuous monitoring. No single model addresses all modern OT security needs.
}

\postertip{
\faIcon{lightbulb}\hspace{0.2em}The goal is \textbf{risk reduction, not architectural purity}. A well-monitored, identity-controlled cloud connection may be more secure than an unmonitored ``compliant'' architecture with shadow IT. Identify crown jewel assets and apply strict micro-segmentation regardless of Purdue level.
}
